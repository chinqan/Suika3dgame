# 物理落下益智遊戲 — QA 測試工程師

**職能**：設計並執行全面的測試計劃，涵蓋功能驗證、效能基準、跨平台測試、邊界條件。

---

## When to Use

- 設定新功能的測試計劃與驗收標準
- 定義效能基準指標（FPS、記憶體、載入時間）
- 建立跨平台/跨瀏覽器測試矩陣
- 測試物理引擎相關的邊界條件
- 回歸測試（修改後確認未破壞現有功能）
- 驗收 GDD 中定義的規格是否正確實作

---

## 🚨 Critical Rules（不可違反）

### GDD vs 程式碼一致性驗證

- **每次 GDD 更新後，必須檢查數值是否與 `config.ts` 一致**
- **將「GDD 數值一致性」納入回歸測試清單** — 物理參數、難度配置、形狀定義表
- **Texture 規格必須可驗證** — 從 GDD 的繪製規格能還原出相同的視覺結果

### 1. 核心玩法測試

```markdown
## 合成機制

- [ ] 兩個同等級形狀碰撞 → 合成為下一等級
- [ ] 最高等級碰撞 → 不合成，正常碰撞
- [ ] 合成位置 → 在兩形狀碰撞中點生成
- [ ] 連鎖合成 → 新形狀可立即觸發下一次合成
- [ ] 跨等級碰撞 → 不觸發合成

## 投放機制

- [ ] 投放後有冷卻期（不可連續快速投放）
- [ ] 投放位置受 X 軸邊界限制
- [ ] 投放冷卻中隱藏瞄準導線

## COMBO 系統

- [ ] 1200ms 內連續合成 → COMBO 計數
- [ ] 超過 1200ms → COMBO 重置
- [ ] COMBO 文字正確顯示（x2, x3...）

## Game Over

- [ ] 形狀靜止後超過 Game Over 線 → 觸發結束
- [ ] 合成瞬間不判定（防止誤判）
- [ ] 下落中形狀不判定
```

### 2. 難度系統測試

```markdown
- [ ] Easy：shapesSlice=2（去掉前 2 種），共 7 等級，maxDropLevel=3
- [ ] Normal：shapesSlice=1（去掉前 1 種），共 8 等級，maxDropLevel=3
- [ ] Hard：shapesSlice=0（全部），共 9 等級，maxDropLevel=4
- [ ] 切換難度 → 正確重置遊戲狀態 + Texture 快取重建
- [ ] 排行榜按難度分組顯示
```

### 3. UI/UX 測試

```markdown
## HUD

- [ ] 分數即時更新
- [ ] NEXT 預覽顯示正確形狀
- [ ] 排行榜/設定按鈕 hover 效果

## Overlay

- [ ] 難度選擇正確顯示
- [ ] Game Over 顯示最終分數與最高分
- [ ] 排行榜正確載入並顯示
- [ ] 設定中音效切換立即生效
- [ ] Player ID 持久化（localStorage）
```

## 效能基準指標

| 指標 | 目標（桌機） | 目標（行動裝置） | 測量方式 |
|------|------------|---------------|---------|
| FPS | ≥ 60fps（穩定） | ≥ 55fps | Stats.js / DevTools |
| 記憶體 | < 100MB | < 80MB | DevTools Memory |
| 首次載入 | < 3s | < 5s | Performance API |
| Texture 快取 | < 20MB | < 10MB | 自訂計數 |
| 同時粒子數 | < 500 | < 200 | Pool activeCount |

### 效能壓力測試場景

1. **最大形狀數** — 容器內塞滿小形狀（~20 個）
2. **快速合成** — 連續觸發 5+ 次合成（粒子 + 音效同時爆發）
3. **長時間運行** — 30 分鐘持續遊戲，監控記憶體成長
4. **最大 COMBO** — 觸發 5+ COMBO，驗證文字/音效不崩潰

### 行動裝置特別測試

- [ ] 不會觸發瀏覽器手勢（pull-to-refresh、swipe back）
- [ ] 橫屏/直屏切換後正確縮放
- [ ] iOS Safari 不出現黑屏

## 邊界條件測試

| 邊界條件 | 預期行為 |
|---------|---------|
| 兩對形狀同時合成 | 各自獨立處理 |
| 形狀卡在牆壁間 | 物理引擎自動推開 |
| 快速連點投放 | 冷卻期內忽略 |
| Player ID 空白 | 使用預設值或匿名 |
| 網路斷線時提交分數 | 靜默失敗，不影響遊戲 |
| localStorage 不可用 | graceful degradation |
| WebGPU 不支援 | 自動 fallback WebGL |

## 工作流程

1. **制定測試計劃** — 根據功能變更確定測試範圍
2. **環境準備** — 確認平台/瀏覽器覆蓋率
3. **功能驗證** — 逐項執行測試清單
4. **效能測量** — 壓力測試 + 基準比對
5. **邊界測試** — 故意觸發極端情況
6. **撰寫報告** — 通過/失敗/風險完整記錄
